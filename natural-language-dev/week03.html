Gemini CLI / GitHub Copilot CLIを使い分ける

Week 3 木曜 1・2限
テーマ
2つのAI CLIを入れて動作原理を比べ、使い分けの方針を立てる
キーワード
Gemini CLI / GitHub Copilot CLI / specialized agents / Plan mode / GEMINI.md / Vibe Coding
必要な準備
進級制作リポジトリがCodespacesで開ける状態・Gemini CLIの認証が完了している状態（第1回完了）・Copilot Student に認証済みの学生は併用可（2026-04-20 以降は新規申請が一時停止中のため、未申請者は Gemini CLI 単体で演習を進める）

この回のゴール

  • Gemini CLI の動作原理——AIエージェント + Built-in Tools（ファイル操作・シェル実行・Web Search・MCP など）で作業するループ——を自分の言葉で説明できる
  • GitHub Copilot CLI の動作原理——AIモデルが specialized agents (Explore / Task / General-purpose / Code-review) に subagent として委譲しながら作業する仕組みと、別軸の Plan mode——を理解する
  • 両CLI を自分の進級制作リポジトリで起動し、同じタスクを投げて挙動の違いを観察する
  • GEMINI.md と Copilot CLI の設定ファイルの役割・置き場所の違いを知る
  • 無料枠・モデル選択の違いを踏まえ、「どのタスクをどちらのCLIに任せるか」使い分けの方針を自分の言葉でメモする
  • Vibe Codingを体験する：自然言語だけで進級制作に小さな改善を加える

前回の振り返り＆2つのCLIの全体像

前回課題の共有（10分）

  • 第2回の docs/context-experiment.md（条件A/B/Cの比較）から、一番差が大きかった発見を1つ口頭共有

なぜ2つのCLIを使い分けるのか（10分）

  • AI CLI は1つに絞れば楽だが、得意分野・料金体系・認証基盤が違う
  • 実務では「このタスクにはこちら、別のタスクにはあちら」と複数ツールを併用するのが普通
  • 今期は Gemini CLI と GitHub Copilot CLI の2本立てで、使い分けの判断を身につける

Gemini CLI の動作原理（10分）

  • Googleが公開しているオープンソースのAIエージェントCLI（公式表記: "open-source AI agent"。google-gemini/gemini-cli）
  • 動作の骨格は AIエージェント + Built-in Tools。Gemini が依頼を読み、必要に応じて組み込みツール（ファイル読取・書込、シェル実行、Web Search、MCP 経由の外部接続 等）を呼び出して作業を進める（Built-in Tools Overview）
  • ※学術的背景の補足: モデルが「考える→ツール呼び出し→結果観察→次の思考…」を繰り返す発想は、ReAct パターン（Yao et al., 2022）の系譜として読める。ただし Gemini CLI 公式は ReAct を名乗っていないので、正しくは「公式表記は『AIエージェント』、裏側の考え方の学術系譜が ReAct」と覚える
  • モデル・無料枠は授業時点の目安として説明。Google公式ページが一次情報なので、詳細はQuota & Pricing のドキュメントで各自確認する習慣を付ける

GitHub Copilot CLI の動作原理（10分）

  • GitHubが提供するエージェントCLI（@github/copilot）。従来の gh copilot 拡張（シェルコマンド補完）とは別物で、こちらはコードベース全体を操作できるエージェント
  • 動作原理の中核: AIモデル自身が判断して、タスクを specialized agent（custom agent）の subagent process に委譲する。公式表現を直訳すると「CLI が使う AI モデルは、得意分野を持つ custom agent の subagent process にタスクを委譲することで作業がより効率的に完了すると判断した場合、そうすることを選べる」（公式docs）。「どの agent を使うか」は人間が選ぶのではなく AI モデル側が判断する点がポイント
  • 組み込みの specialized agents は4つ: Explore / Task / General-purpose / Code-review。用途別に得意分野が違い、メインモデルが必要に応じて呼び出す
  • 別軸で Plan mode がある。agent ではなく対話モードのスイッチで、コード変更を実行する前に計画を提示してレビューさせる仕組み。起動は Shift+Tab キーボードショートカット（オン/オフのトグル）
  • 無料枠は Copilot のプランに依存（Copilot Student に認証済みなら在学中は無償で Copilot Pro 相当の機能が使える。ただし 2026-04-20 以降は新規申請が一時停止中で、再開時期は未定）。詳細はCopilot プランの最新案内を参照

両CLIを Codespaces に導入して動かす

Gemini CLI の動作確認（10分）

  • 第1回で認証済みの前提。Codespacesのターミナルで npx @google/gemini-cli を起動し、進級制作リポジトリに cd した状態で @README.md この構成を3行で説明 等を試す
  • 認証が切れていたら再認証（Googleアカウントのブラウザ認証）

GitHub Copilot CLI の導入（25分）

  • Copilot Student の状態確認——GitHub Student Developer Pack 経由で Copilot が有効になっているか確認。新規申請は 2026-04-20 以降一時停止中のため、未申請者は今日は Gemini CLI 単体で演習を進めてOK（再開時期は授業冒頭で都度アナウンス）
  • Codespacesのターミナルで npm install -g @github/copilot（または公式ドキュメントの最新インストール手順）
  • 初回起動時に GitHub アカウントの認証フローを完了
  • 動作確認: copilot を起動し、プロンプト内で ?（またはターミナル側で copilot help）を実行してスラッシュコマンドの一覧を確認。Plan mode は Shift+Tab でオン/オフを切り替えられる

設定ファイルの違いを見る（15分）

  • 進級制作リポジトリのルートに、最小の GEMINI.md を作成（プロジェクトの目的・構成・守ってほしい規約を3〜5行でメモ）
  • Copilot CLI 側の設定は別ファイルで管理される——プロジェクトごとの指示は公式の指示ファイル（命名は公式ドキュメントに従う）に書く
  • 「同じ "プロジェクトのメモ" でも、CLIが違えばファイルも違う」ことを実感。後で自動同期の仕組みを考えるための伏線
導入が詰まったら: 「Gemini CLI は動いたが Copilot CLI が詰まった（あるいは逆）」という状況は普通に起こります。今日は少なくとも片方が動く状態でこの先の演習に入れればOK。両方動かなかった人は講師に声をかけてください。

同じタスクを両CLIに投げて比べる

今日の問い: 同じ依頼でも、CLIが違えば「考え方」「出力」「手数」がどれくらい違うのか？

共通のお題（例）

  • 自分の進級制作から1つ機能を選び、それぞれのCLIに この機能の主要なファイルを洗い出し、役割を200字程度で要約して と依頼する
  • 片方でしか動かない人は、動く方だけで実施してよい（比較は近くの人の画面を見せてもらう）

実験A: Gemini CLI でやる（15分）

  • 依頼文の末尾に「どのファイルを読んだか列挙してから要約して」と付け、ツール呼び出しの足跡を見る
  • 何回ファイルを読みに行ったか、どんなツールを使ったかをメモ

実験B: Copilot CLI でやる（15分、Copilot Student 未認証の人は近くの人の画面を見る）

  • Shift+Tab で Plan mode に切り替え、依頼を投げて実行前の計画を見せてもらう
  • 計画を承認してから実行。Explore や Task 等の specialized agents がどのように呼び出されるか観察

観察比較（20分）

  • 「実行前に計画を見せる／見せない」の違いがUXにどう効くか
  • ツール呼び出しのトレース（Gemini CLI）と specialized agents の役割分担（Copilot CLI）、どちらが自分には分かりやすかったか
  • 出力の質を主観で比較（事実誤認の有無、粒度の粗さ細かさ）——評価は自分がする。AIに「どっちが良いですか？」は聞かない
観察ポイント: 「どちらが優秀か」ではなく、「自分のこのタスクにどう合う／合わないか」。使い分けの方針は相対評価で決まります。

Vibe Coding：自然言語だけで小さな改善を入れる

Vibe Coding とは（5分）

  • Andrej Karpathy が提唱した言葉（2026年初頭）。自然言語で指示し、AIが書いたコードをほぼそのまま採用する開発スタイル
  • 「雑でいい、動けばいい」の体験——後の回（第4回）でVibe Coding の限界を扱うための下地

実践（20分）

  • 進級制作に、小さな改善を1つ選ぶ（例: 文言の修正、軽微なUIの整理、関数名のリネーム、READMEへの使い方追記、小さなバリデーション追加 等）
  • 選んだCLI（どちらでも可）に自然言語だけで依頼——コードは自分で書かない
  • 変更が妥当なら git add && git commit。不安なら git stash で一旦置く

メモ（5分）

  • どのCLIを使ったか／なぜそちらを選んだか
  • AIが書いたコードをそのまま採用したか、どこを手直ししたか
まとめ 約10分
